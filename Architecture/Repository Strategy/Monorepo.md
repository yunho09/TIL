# 모노레포(Monorepo)
## 정의

- **Monolithic Repository**의 약자이다.
- **하나의 저장소(Repository)** 안에서 **여러 개의 프로젝트**를 통합하여 관리하는 소프트웨어 개발 전략이다.
- 반대 개념은 **Multirepo(멀티레포)** 또는 **Polyrepo**라고 한다.

## 비교
| **비교 항목**  | **Multirepo (멀티레포)**        | **Monorepo (모노레포)**       |
| ---------- | --------------------------- | ------------------------- |
| **구조**     | 1 저장소 = 1 프로젝트              | 1 저장소 = N 프로젝트            |
| **코드 공유**  | 어렵다 (npm 배포 후 재설치 필요)       | **쉽다** (import로 바로 가져옴)   |
| **관리 포인트** | 프로젝트마다 설정(ESLint 등) 따로 해야 함 | **한 번만 설정**하면 모두 적용됨      |
| **버전 관리**  | 프로젝트마다 버전이 제각각임             | 통합된 버전 관리가 가능함            |
| **빌드 속도**  | 빠름 (내 것만 빌드하면 됨)            | 느려질 수 있음 (도구가 없으면 전체 빌드됨) |
## 장점과 단점
### 장점

1. **코드 재사용성 극대화:**
    
    - `Button`, `Input` 같은 공통 UI 컴포넌트나 유틸리티 함수를 하나의 폴더에 두고 여러 프로젝트에서 바로 갖다 쓸 수 있다.
    
2. **의존성 관리 용이:**
    
    - 모든 프로젝트가 `React` 버전을 똑같이 맞추거나 설정 파일(ESLint, Prettier)을 하나로 통일하기 편하다.
     
3. **Atomic Commit:**
    
    - API가 바뀌어서 프론트엔드와 백엔드를 동시에 고쳐야 할 때, 한 번의 커밋으로 두 프로젝트를 동시에 수정할 수 있다.
    

### 단점

1. **저장소 용량 비대화:**
    
    - 모든 코드가 한곳에 있다 보니 `git clone` 할 때 엄청 오래 걸릴 수 있다.
        
2. **빌드 복잡도 증가:**
    
    - 코드 하나 고쳤는데 전체 프로젝트를 다 다시 빌드하면 시간이 너무 오래 걸린다.
        
    - 그래서 **Nx, Turborepo** 같은 똑똑한 빌드 도구가 필수다.
        
3. **접근 제어 불가:**
    
    - 신입 개발자에게 특정 프로젝트만 보여주고 싶은데, 저장소를 복제하면 모든 코드가 다 보인다.

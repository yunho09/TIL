# SOLID 원칙

## 정의

- 객체 지향 설계에서 시스템을 더 이해하기 쉽고 유연하며 유지보수하기 좋게 만들기 위한 5가지 기본 지침

## 종류

#### 1. SRP: 단일 책임 원칙 (Single Responsibility Principle)

- "하나의 클래스는 하나의 책임만 가져야 한다."
	
	- **설명:** 클래스가 여러 가지 일을 도맡아 하면 한 기능을 수정할 때 상관없는 다른 기능까지 망가질 위험이 크다.
    
	- **효과:** 코드 응집도 상승 및 수정 시 영향 범위 최소화

#### 2. OCP: 개방-폐쇄 원칙 (Open-Closed Principle)

- "확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다."
	
	- **설명:** 기존의 코드를 변경하지 않으면서도 기능을 추가할 수 있도록 설계해야 한다는 원칙
    
	- **효과:** 시스템의 안정성 유지

#### 3. LSP: 리스코프 치환 원칙 (Liskov Substitution Principle)

- "자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다."
	
	- **설명:** 부모 클래스 자리에 자식 클래스를 넣어도 원래 의도대로 동작해야 함
    
	- **효과:** 다형성을 안전하게 활용 상속 관계 견고

#### 4. ISP: 인터페이스 분리 원칙 (Interface Segregation Principle)

- "자신이 사용하지 않는 메서드에 의존하도록 강제해서는 안 된다."
	
	- **설명:** 하나의 거대한 인터페이스보다 여러 개의 구체적인 인터페이스가 낫다. 클래스가 꼭 필요한 메서드만 구현하도록 인터페이스를 작게 쪼개야 한다.
    
	- **효과:** 특정 인터페이스의 변화가 해당 기능을 사용하지 않는 다른 클래스에 영향 X

#### 5. DIP: 의존역전 원칙 (Dependency Inversion Principle)

- **"추상화에 의존해야지 구체화에 의존하면 안 된다."**
	
	- **설명:** 구현체(Class)보다는 역할(Interface)에 집중하라는 의미
    
	- **효과:** 구현체를 자유롭게 변경 가능

--- 

| 원칙      | 요약              | 키워드        |
| :------ | :-------------- | :--------- |
| **SRP** | 한 클래스는 한 가지 일만  | 책임 분리      |
| **OCP** | 수정은 No, 확장은 Yes | 인터페이스, 추상화 |
| **LSP** | 자식은 부모의 규약을 준수  | 타입 호환성     |
| **ISP** | 필요한 인터페이스만 구현   | 인터페이스 세분화  |
| **DIP** | 클래스보다 인터페이스에 의존 | 의존성 주입     |
